# ניתוח מעמיק + בריף עיצוב — Troubleshooting (Playbooks)

> **מטרה:** ניתוח מלא של אשכול פתרון הבעיות, ובריף עיצוב לשחזור ויזואלי ברמה הגבוהה ביותר.
> **תפקיד (מ-PRD §11):** `Troubleshooting` (ספרייה), `FlowEditor` (עורך), `FlowPlayer` (נגן), `SharedGuide` (שיתוף ציבורי).
> **מקורות:** PRD §3.7, §4.3, §5.5 · SRS §1.8.

---

## 0. שם — המלצה

"Playbook" שאתה משתמש בו — מצוין. מודרני וברור יותר מ"תסריט". אמץ אותו בממשק.
בקוד, הישות נשארת `TroubleshootingFlow` (לא לשנות את הסכמה).

---

## 1. הניתוח — Playbook הוא גרף, לא רשימה

בניגוד לשיעור (לינארי, עמוד אחרי עמוד), **Playbook הוא עץ החלטות מסועף**:
הנציג מתחיל בנקודת כניסה, עונה על שאלות, ולפי התשובות מנותב למסלולים שונים
עד שמגיע לפתרון. המבנה הבסיסי (`flow_data`) הוא שני מערכים:

- **`nodes[]`** — הצמתים (השאלות, הפעולות, הפתרונות).
- **`connections[]`** — הקשתות שמחברות ביניהם (`sourceNodeId → targetNodeId`,
  עם `optionIndex` ו-`label`).

זה משנה את כל גישת העיצוב: **צמתים וחיבורים**, לא עמודים.

---

## 2. שש סוגי הצמתים + שפת הצבעים (ליבה ויזואלית — משותפת לעורך ולנגן)

לכל צומת: `id, type, title, description, note, position{x,y}, media[], nextNodeId`.
זו שפת הצבעים המוצעת — מתואמת למותג, ותשמש **גם בעורך וגם בנגן** לעקביות:

| סוג | תפקיד | צבע מוצע | אייקון (outline) |
|---|---|---|---|
| `start` | נקודת כניסה (אחת לכל Playbook) | כחול כהה #0075DB | דגל / play |
| `question` | צומת החלטה — מציג `options[]` | כחול מותג #2EB4FF | שאלה / הסתעפות |
| `action` | "בצע פעולה" — מציג `actions[]` | ענבר #F59E0B | רשימת-משימות |
| `solution` | פתרון — סיום מוצלח | ירוק success #22C55E | V במעגל |
| `end` | סיום (למשל הסלמה) | אפור #757D86 | עצירה |
| `linked_flow` | קפיצה ל-Playbook אחר | סגול #8B5CF6 | קישור חיצוני |

**מבני-משנה חשובים:**
- `options[]` (בצמתי question): `{id, text, note, targetNodeId, linkedFlowId,
  linkedTargetNodeId, hyperlink{url,label}, media[]}` — כל אפשרות מנווטת ליעד.
- `actions[]` (בצמתי action): `{text, note, linkedFlowId, linkedTargetNodeId,
  hyperlink{url,label}, media[]}`.
- `media[]`: `{id, type(image/gif/video), url, alt, size}` — לכל צומת/אפשרות.

---

## 3. Deep-linking — הגרף שמעבר ל-Playbook הבודד

אופציה או פעולה יכולות להפנות לצומת ב-**Playbook אחר** (`linkedFlowId` +
`linkedTargetNodeId`). המשמעות: ה-Playbooks מתחברים לגרף-ידע משותף, לא עצים
מבודדים. ויזואלית זה דורש טיפול מיוחד — צומת/תגית "קישור חיצוני" (סגול) שמסמן
"מכאן קופצים ל-Playbook אחר".

---

## 4. שלושת המסכים — ניתוח

### 4.1 הספרייה (`Troubleshooting`) — נקודת הכניסה
מי: כולם (עריכה: instructor+). מה: קטלוג כל ה-Playbooks + לשונית "תסריטים חסרים".
- כרטיס Playbook: כותרת, תיאור, תגית קטגוריה (8 קטגוריות), רמת קושי
  (קל/בינוני/מתקדם), `usage_count`, `success_rate`, `avg_completion_time`, תגיות.
- חיפוש + סינון (קטגוריה / קושי / תגית).
- **שתי לשוניות:** "ספריית Playbooks" + "תסריטים חסרים" (מ-`TroubleshootingSession`
  עם `missing_flow=true`, עם דגל `handled` לסימון טיפול).
- כפתור יצירה (instructor+).

### 4.2 העורך (`FlowEditor`) — כלי הדסקטופ
מי: instructor+. מבוסס **@xyflow/react** (React Flow). האישיות: כלי מקצועי וצפוף
(חשוב על n8n / Figma).
- קנבס עם צמתים ממוקמים לפי `position{x,y}` וחיבורים כקשתות.
- פאנל צד: פלטת צמתים (6 הסוגים) + מאפייני הצומת הנבחר (כותרת, תיאור, מדיה,
  options/actions, deep-links).
- צמתים מותאמים צבע-מקודדים לפי שפת הצבעים (§2).
- סרגל כלים: שמירה (יוצרת `version` + רשומת `version_history`), היסטוריית גרסאות,
  הרשאות עריכה (`edit_permissions`), שיתוף (`share_settings`), פרסום
  (`is_published`), ו-**עוזר AI** (`troubleshooting_flow_builder` — מזהה צמתים
  מנותקים, מציע גשרים, מחזיר JSON מובנה).
- minimap + בקרות זום.
- **המסך המורכב ביותר — יקבל בריף ייעודי.**

### 4.3 הנגן (`FlowPlayer`) — חוויית השטח
מי: כולם. **התובנה הקריטית: mobile-first.** המשתמשים הם טכנאי שטח, בשטח, על
הטלפון, תוך כדי שיחה עם לקוח. אישיות מנוגדת לעורך: רגוע, ממוקד, מהיר, מותאם מגע.
- מציג צומת אחד בכל רגע: כותרת, תיאור, מדיה.
- צומת question: האפשרויות ככפתורים גדולים → ניווט ל-`targetNodeId`.
- צומת action: צ'קליסט פעולות + "המשך".
- solution/end: מסך סיום.
- רישום שקוף ל-`session_log[]`; בסיום — טופס משוב (`FlowFeedback`:
  `was_helpful`, `rating`, `customer_sentiment`, `resolved_at_step`, הצעות).
- ניווט "חזרה" לצעדים קודמים.
- **יקבל בריף ייעודי.**

### 4.4 שיתוף ציבורי (`SharedGuide`)
`SharedGuideLink` (token, public_url, view_count) — תצוגה ציבורית של מדריך דרך
טוקן, ללא התחברות. נטפל בו עם הנגן.

---

## 5. רצף בנייה מומלץ

1. **הספרייה** (להלן — מוכן). נקודת הכניסה, מגדירה את כרטיס ה-Playbook.
2. **הנגן** (בריף הבא). חוויית הקצה, mobile-first — מגדיר איך צומת "נראה" למשתמש.
3. **העורך** (בריף אחרון). הכי מורכב — יושב על שפת הצבעים ועל מבנה הצומת שכבר נסגרו.

## 📋 בריף #1 — ספריית ה-Playbooks (העתק מכאן)

```
אני רוצה לעצב את ספריית פתרון הבעיות (Playbooks) של iPracticom Academy.
חשוב מאוד: השתמש ברכיבים, בטוקנים ובשפת העיצוב הקיימים בפרויקט (דשבורד,
הכשרות, תוכן הכשרה) — עקביות מלאה, אל תיצור שפה חדשה.

— זהות מותג (לאכיפה) —
• עברית מלאה, RTL.
• צבעים: ראשי #2EB4FF, כחול כהה #0075DB, אפור #757D86, כהה #181D24,
  רקע בהיר #F4FBFF.
• שפה: tech-human, נקייה, אייקוני outline, אלמנטים גאומטריים, אנימציות עדינות.
• לוגו שמאל-עליון ב-50% שקיפות. פוטר: © iPracticom Academy - לשימוש פנימי בלבד

— מהות המסך —
ספריית Playbooks לפתרון בעיות בשטח. המטרה: שטכנאי ימצא במהירות את ה-Playbook
הרלוונטי לתקלה שמולו. בנה רכיב לשימוש חוזר PlaybookCard ו-layout בשם
PlaybooksLibrary.

— מבנה הדף —
1) כותרת "פתרון בעיות" + תת-כותרת + כפתור "Playbook חדש" (לבעלי הרשאה).
2) שורת חיפוש + פילטרים: קטגוריה, רמת קושי, תגית.
3) שתי לשוניות: "ספריית Playbooks" (ברירת מחדל) ו-"תסריטים חסרים".

— כרטיס Playbook (PlaybookCard) —
• תגית קטגוריה צבעונית (אחת מ-8: מרכזיות ענן PBX / Firewall MikroTik /
  מצלמות אבטחה / מערכות סאונד / גילוי אש / בקרי טמפרטורה / רשתות / כללי).
• כותרת + תיאור קצר.
• מטא בשורה: רמת קושי (קל/בינוני/מתקדם), זמן השלמה ממוצע (דקות),
  שיעור הצלחה (%), מספר שימושים.
• תגיות.
• בלחיצה → נפתח הנגן.
• למשתמש עם הרשאת עריכה: אייקון עריכה קטן בכרטיס → העורך.

— לשונית "תסריטים חסרים" —
טבלה/רשימה של שיחות שירות שבהן לא נמצא Playbook: שם הנציג, מספר טלפון,
תיאור התקלה החסרה, משך, ודגל "טופל". כפתור "צור Playbook מתקלה זו".

— מצבים —
• מאוכלס (ברירת מחדל) — נתוני דמו בעברית (ראו למטה).
• ריק — אין Playbooks: מצב ריק עם איור גאומטרי + "עדיין אין Playbooks".
• מובייל — כרטיסים בעמודה אחת, פילטרים מתקפלים.

— נתוני דמו —
• "אין צליל חיוג במרכזיית PBX" — מרכזיות ענן, בינוני, 4 דק', 92% הצלחה, 47 שימושים.
• "מצלמת אבטחה לא מתחברת לרשת" — מצלמות אבטחה, קל, 3 דק', 88%, 134 שימושים.
• "חסימת תעבורה ב-MikroTik Firewall" — Firewall MikroTik, מתקדם, 9 דק', 76%, 23.
• "גלאי עשן בהתראת שווא" — גילוי אש, בינוני, 6 דק', 81%, 58.

— אילוצים טכניים —
React + Tailwind + shadcn/ui, אייקוני lucide-react (outline),
אנימציות עדינות (framer-motion). שמור על רכיבים קיימים — אל תפשט/תמיר.

תתחיל מהלשונית המאוכלסת בדסקטופ, ואחר כך נדייק.
```

## אנטומיית הכרטיס — רפרנס (שדות SRS §1.8)

| אלמנט | שדה |
|---|---|
| כותרת / תיאור | `TroubleshootingFlow.title` / `description` |
| קטגוריה | `category` (enum, 8 ערכים) |
| רמת קושי | `difficulty_level` (קל/בינוני/מתקדם) |
| זמן ממוצע | `avg_completion_time` (דקות) |
| שיעור הצלחה | `success_rate` |
| שימושים | `usage_count` |
| תגיות | `tags` |
| פורסם | `is_published` |
| לשונית חסרים | `TroubleshootingSession` (`missing_flow`, `handled`, `agent_name`, `phone_number`, `missing_flow_description`) |
