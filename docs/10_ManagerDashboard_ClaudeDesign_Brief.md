# בריף עיצוב — דשבורד ניהול (ManagerDashboard)

> **מטרה:** דשבורד אנליטיקה למנהל מחלקה — נתוני עובדים, השוואות, וייצוא דוחות.
> **תפקיד (PRD §11):** `ManagerDashboard` — manager+. (CRUD משתמשים: `UserManagement`, נפרד.)
> **מקורות:** PRD §4.2, §11 · SRS §1.1 (User: `managed_department`, `department`, `progress_stats`).

---

## 1. מודל הגישה (קריטי)

מנהל מנהל **מחלקה** (`User.managed_department`). "העובדים שלו" = כל המשתמשים
ש-`department` שלהם = `managed_department` שלו. **כל הנתונים מסוננים למחלקה
שלו** (admin רואה את כל המחלקות). אכיפה ב-RLS/בפונקציה — לא רק ב-UI.

---

## 2. מיפוי הבקשות → נתונים

| בקשה | מקור | סטטוס |
|---|---|---|
| נתונים פר-עובד | `progress_stats` של כל עובד + `ExamAttempt` + `UserCertificate` | ✓ מעוגן |
| השוואות מחלקתיות | צבירה של `progress_stats` על עובדי המחלקה | נתונים מעוגנים, פונקציות צבירה חדשות |
| ייצוא דוחות | דפוס קיים `exportWorkHours` → פונקציה חדשה לצוות | חדש (דפוס קיים) |
| drill-down לעובד | **שימוש חוזר ברכיבי הפרופיל (מסמך 09)** | ✓ |

---

## 3. המלצות נוספות (מה עוד) — מעוגן / חדש

**מעוגן:**
- **עובדים בסיכון:** מי שמפגר (`avg_progress` נמוך), לא פעיל (`last_activity` ישן),
  או נכשל במבחנים. (מתחבר ל-"Learner At-Risk Detector" שברודמ\פ.)
- **סקירת פעילות שבועית:** `weekly_lessons`, `weekly_time_spent`, `last_activity`.
- **מעקב דדליינים:** סטטוס מול תאריכי יעד (אוטומציית `checkTrackDeadlines` קיימת).
- **משפך השלמות:** כמה במחלקה השלימו כל מסלול/מודול.
- **KPIs מחלקתיים:** התקדמות ממוצעת, אחוז השלמה, ציון מבחן ממוצע, תעודות,
  לומדים פעילים.

**חדש (לעתיד):**
- התראות אוטומטיות על עובד בסיכון (אוטומציה).
- דוח למידה שבועי אוטומטי במייל (ברודמ\פ).
- השוואה ל-benchmark חברה-רחב (admin).

---

## 4. פריסת העמוד

1) **כותרת:** "דשבורד ניהול — מחלקת [X]" + בורר תקופה + כפתור "ייצוא דוח".
2) **כרטיסי KPI מחלקתיים** (שורה עליונה): התקדמות ממוצעת, אחוז השלמה, ציון מבחן
   ממוצע, לומדים פעילים, תעודות, מספר עובדים בסיכון.
3) **השוואת עובדים:** bar chart / דירוג של עובדים לפי מדד נבחר (התקדמות / ציון /
   XP / זמן).
4) **עובדים בסיכון:** פאנל עם העובדים שדורשים תשומת לב + סיבה.
5) **טבלת הצוות:** כל עובדי המחלקה — שם, מסלול מוקצה, התקדמות %, ציון ממוצע,
   פעילות אחרונה, סטטוס. ניתנת למיון; לחיצה → drill-down.
6) **drill-down עובד** (פאנל/מודאל): פרופיל מלא של העובד בשימוש חוזר ברכיבי
   מסמך 09 (התקדמות, מבחנים, תעודות, פעילות).

## 📋 הפרומפט — העתק מכאן

```
אני רוצה לעצב דשבורד ניהול (ManagerDashboard) ל-iPracticom Academy — תצוגת
מנהל מחלקה לאנליטיקה והשוואת עובדים. השתמש בשפת העיצוב והטוקנים הקיימים
בפרויקט — עקביות מלאה, RTL.

— מותג: עברית RTL · #2EB4FF / #0075DB / #757D86 / #181D24 / רקע #F4FBFF ·
  tech-human · אייקוני outline · אנימציות עדינות · לוגו שמאל-עליון 50% · פוטר.

— מהות: דשבורד אנליטי (לא דף למידה). מנהל רואה רק את עובדי המחלקה שלו.

— מבנה —
1) כותרת: "דשבורד ניהול — מחלקת תמיכה טכנית" + בורר תקופה (שבוע/חודש/הכל) +
   כפתור "ייצוא דוח".

2) כרטיסי KPI מחלקתיים (שורה): התקדמות ממוצעת, אחוז השלמה, ציון מבחן ממוצע,
   לומדים פעילים השבוע, סך תעודות, ומספר "עובדים בסיכון" (מודגש).

3) השוואת עובדים: bar chart אופקי שמדרג את העובדים לפי מדד, עם בורר מדד
   (התקדמות / ציון ממוצע / XP / זמן למידה).

4) פאנל "עובדים בסיכון": רשימת עובדים הדורשים תשומת לב, כל אחד עם הסיבה
   ("לא פעיל 12 ימים" / "התקדמות 18%" / "נכשל ב-2 מבחנים") וכפתור "צפה".

5) טבלת הצוות: כל עובדי המחלקה — עמודות: שם+אווטאר, מסלול מוקצה, התקדמות (בר %),
   ציון ממוצע, פעילות אחרונה, סטטוס (פעיל/בסיכון/הושלם). מיון לפי כל עמודה.
   לחיצה על שורה → drill-down.

6) drill-down עובד (פאנל צד/מודאל): פרופיל מלא של העובד — התקדמות, היסטוריית
   מבחנים, תעודות, פעילות. (שימוש חוזר ברכיבי עמוד הפרופיל.)

— נתוני דמו: מחלקת "תמיכה טכנית", 7 עובדים בעברית עם נתונים מגוונים, 2 מהם
  בסיכון. השתמש ב-bar chart להשוואה ובברי התקדמות בטבלה.

— מצבים: דסקטופ-first (כלי אנליטי). הראה גם מובייל מצומצם (KPIs + רשימה).
— טכני: React + Tailwind + shadcn/ui, lucide-react (outline), recharts לגרפים,
  framer-motion. שמור על רכיבים קיימים.

תתחיל מהמבט המלא בדסקטופ (KPIs + השוואה + טבלת צוות), ואחר כך נדייק drill-down.
```

## אנטומיה — רפרנס שדות

| אזור | שדות |
|---|---|
| סינון מחלקה | `User.managed_department` = `User.department` |
| KPI מחלקתי | aggregate של `progress_stats.{avg_progress, exams_passed, avg_score, certificates_earned}` |
| השוואת עובדים | `progress_stats.{avg_progress, avg_score, total_xp, total_time_spent_minutes}` פר-עובד |
| בסיכון | נגזר: `avg_progress` נמוך / `last_activity` ישן / מבחנים שנכשלו |
| טבלת צוות | `User.{full_name, profile_image_url, assigned_track_id, department}` + `progress_stats` |
| drill-down | כל מקורות עמוד הפרופיל (מסמך 09) |

## פונקציות לבנייה (Claude Code)

- `getDepartmentStats(managed_department)` → KPIs מצטברים (חדש).
- `getDepartmentMembers(managed_department)` → רשימת עובדים + stats (חדש, scoped).
- `getAtRiskLearners(managed_department)` → עובדים בסיכון לפי כללים (חדש).
- `exportTeamProgress({managed_department, filters, format})` → CSV/PDF (חדש,
  על דפוס `exportWorkHours`).
- **RLS:** כל הפונקציות מסוננות ל-`managed_department`; admin עוקף.

## עמוד נלווה (לעתיד)

`UserManagement` (manager+) — ה-CRUD של המשתמשים (הזמנה, תפקיד, סטטוס, שיוך
מסלול). נפרד מהאנליטיקה. בריף משלו כשנגיע אליו.

*תצוגת המנהל מתחילה כאן. הבאים בתור (אם תרצה): UserManagement, ותצוגת ה-admin המלאה.*
