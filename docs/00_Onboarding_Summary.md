# סיכום קליטה — iPracticom Academy

> **מטרה:** סיכום ההבנה של הארכיטקטורה, תוכנית העבודה, ואופן העבודה — כפי שנקרא
> משלושת מסמכי מקור-האמת (36, 37, 35) לפני תחילת הבנייה.
> **מעמד:** מסמך התמצאות פנימי. מקורות-האמת עצמם: 36 + 37 (ארכיטקטורה) > 35 (נתונים).

---

## 1. הארכיטקטורה

- אנחנו בונים רק את השכבות העליונות: **Frontend (React/Vite/TS)** + **לוגיקה עסקית**
  + **API Client מופשט** (`apiClient.*`).
- מסד הנתונים (MySQL בענן החברה), ה-API, ה-auth והאחסון = **באחריות צוות הפיתוח**.
  אנחנו לא נוגעים בהם.
- **הלב של ההפרדה:** כל גישה לנתונים עוברת דרך interface מופשט. עד קבלת ה-API
  האמיתי — מימוש `MockApi` שקורא מ-`data/app-backup` (5,729 רשומות אמיתיות).
  המעבר mock→real בסוף = החלפת factory בלבד, בלי לגעת בלוגיקה.
- **Auth נדחה ל-Phase 12.** בפיתוח: mock user. מסכי ההתחברות מחכים מעוצבים ולא-פעילים.
- **היעד:** אפליקציית web מלאה שרצה על localhost מול mock עם הדאטה האמיתי, ואז
  מסירה לצוות הפיתוח לחיבור ה-API והטעינה ל-DB.

```
┌─────────────────────────────────────────┐
│  Frontend (React) — כל המסכים            │  ← אנחנו
│  Business Logic + Services               │  ← אנחנו
│    ↓ דרך ↓                                │
│  API Client (interface מופשט)            │  ← אנחנו (חוזה)
├─────────────────────────────────────────┤
│  Internal API (MySQL + Auth + Storage)   │  ⛔ צוות הפיתוח
└─────────────────────────────────────────┘
```

יחידה `recalculateUserStats` — אף מסך לא מחשב לבד.

## 3. אופן העבודה (כללים מחייבים)

1. **שלב אחד בכל פעם.** לא מתחילים חדש עד שהקודם עובד ואושר.
2. לכל שלב — **קוראים קודם** את הבריף הרלוונטי (02-33) + מסך ה-HTML מ-`design-export`.
   לא מנחשים, לא ממציאים. אם חסר מידע — שואלים.
3. **משתמשים במסכי ה-HTML כבסיס** — לא מעצבים מחדש, מחברים אליהם לוגיקה.
4. **כל הגישה לנתונים דרך `apiClient.*`** — לעולם לא HTTP ישיר או DB.
5. **קוד קצר:** קובץ ≤150 שורות, אחריות אחת.
6. **שמות שדות לפי הדאטה האמיתי** (מסמך 35): `role` (לא `system_role`),
   `created_date`, `created_by_id`, `managed_department`.
7. **מקור-אמת יחיד** לכל נתון — במיוחד התקדמות משתמשים (Phase 1).
8. **אין secrets בקוד** — רק מ-env.
9. **סדר עדיפויות בסתירה:** 36/37 (ארכיטקטורה) > 35 (נתונים) > 34 (reference) >
   02-33 (עיצוב) > 01 (היסטוריה).

**מדריך קריאה לכל שלב** (מסמך 37 §6): הפרומפט ממסמך 36 → מסך ה-HTML מ-`design-export`
→ הבריף הרלוונטי (02-33) → מסמך 35 לשמות שדות ומבנה נתונים.

## 4. הדאטה האמיתי (מסמך 35)

**19 ישויות, 5,729 רשומות** (כולל User). נפחים עיקריים:

| ישות | רשומות | | ישות | רשומות |
|---|---|---|---|---|
| UserProgress | 5,000 | | Exam | 17 |
| Question | 317 | | WizardConfig | 17 |
| Concept | 96 | | SharedModule | 11 |
| ModuleLesson | 89 | | CandidateAssessment | 10 |
| Invite | 56 | | AppSetting | 6 |
| Topic | 39 | | LearningTrack | 3 |
| TroubleshootingFlow | 23 | | RoleUpgradeRequest | 2 |
| TrackModule | 22 | | User | 21 |

- **ריקות — לדלג:** Course, KnowledgeArticle, ModuleExam (0 רשומות).
- **מזהים = Mongo ObjectIDs** → נשמרים as-is כ-`id TEXT PRIMARY KEY`.
- **שדות jsonb** (`blocks`, `flow_data`, `design`, `value`) → ייבוא as-is.
- **התוכן קטן** → מיגרציה פשוטה ומהירה. עיקר הנפח = 5,000 אירועי UserProgress.
- **קריאת שדות מהדאטה** (`app-backup`), לא מ-`schema-export` (שהוא תרשים יחסים בלבד).

**מיפוי שדות** (בכל טבלה):
```
_id / id       → id (TEXT, as-is)
created_date   → created_at   (שם היעד נקבע בשלב ה-DDL; במקור: created_date)
updated_date   → updated_at
created_by_id  → created_by   (soft ref עד מיפוי users)
role           → role         (לא system_role)
jsonb          → as-is
```

## 5. נקודות לתשומת לב / לאישור

1. **מנהל לפי `managed_department`, לא לפי `role`** (§6.3) — ה-enum תומך
   ב-manager/instructor אבל בפועל הדאטה משתמש רק ב-`user`(18)/`admin`(3).
   אכיפת ManagerDashboard תיבדק לפי "יש `managed_department` מוגדר" (5 משתמשים).
2. **שמות יעד ב-Supabase סגנון** (`auth_uid`, `profiles`, `created_at`) מופיעים
   במסמך 35 מהמעבר הקודם — אבל ההחלטה הסופית (36/37) היא **MySQL+API**. שמות היעד
   הסופיים נקבעים בשלב ה-DDL (2.2); בינתיים בשכבת ה-mock קוראים את השמות הגולמיים
   (`created_date` וכו').
3. **הלוגו — נמצא ✅** ב-`assests/IP-COM Design System_files/ipracticom-logo.svg`.
   wordmark של "ipracticom", 109×16.4, עם gradient מ-`#282FEF` ל-`#33B1FF`. מוטמע
   כ-inline SVG בשלב 0.5 (AppShell). (מסמך 28 עצמו לא קיים ב-`docs/` — הקובץ הוא המקור.)
4. **קובץ הדאטה:** `app-backup-2026-06-29.json`. מסמך 35 §6 מזכיר ייצוא מעודכן
   (16:32) עם User → מניחים שהקובץ הזה הוא המעודכן עם 19 הישויות.
