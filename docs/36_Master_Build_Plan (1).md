## 🏛️ הארכיטקטורה (החלטה סופית — קרא קודם)

הצוות עובד עם **MySQL בענן החברה + שכבת API פנים-ארגונית** שצוות הפיתוח כותב.
**אנחנו לא נוגעים במסד הנתונים ולא ב-auth.** Claude Code מדבר רק מול ה-API.

```
┌─────────────────────────────────────────┐
│  Frontend (React) — כל המסכים            │  ← אנחנו
│  Business Logic + Services               │  ← אנחנו
│    ↓ דרך ↓                                │
│  API Client (interface מופשט)            │  ← אנחנו (חוזה)
├─────────────────────────────────────────┤
│  Internal API (MySQL + Auth + Storage)   │  ⛔ צוות הפיתוח
└─────────────────────────────────────────┘
```

**המטרה:** אפליקציית web מלאה שרצה על **localhost** מול **mock עם הדאטה האמיתי**,
כל הלוגיקה כתובה ונקייה. בסוף — מוסרים לראש הצוות, מקבלים מפתח API, ומחברים.

## עקרונות-על (חלים על כל שלב)

1. **שכבת API Client:** הלוגיקה מדברת מול interface מופשט (`apiClient.*`), לא מול
   DB ולא מול HTTP ישיר. עד קבלת ה-API — mock עם הדאטה האמיתי. החלפה בסוף = שכבה
   אחת בלבד, בלי לגעת בלוגיקה.
2. **Auth נדחה לסוף** (Phase 12). בפיתוח: mock user. מסכי ההתחברות מחכים מוכנים.
3. **אחסון דרך ה-API** (`apiClient.uploadFile`) → שרת החברה. לא R2/Supabase.
4. **שלב = משימה אחת.** לא מתחילים חדש עד שהקודם עובד + בדיקת שפיות.
5. **קוד קצר:** קובץ ≤150 שורות, אחריות אחת.
6. **מקור-אמת יחיד** לכל נתון (במיוחד התקדמות).
7. **שמות שדות לפי הדאטה האמיתי** (מסמך 35): `role` (לא system_role),
   `created_date`, `created_by_id`, `managed_department`.
8. בכל פרומפט — תן ל-Claude Code את המסמכים הרלוונטיים לקרוא **לפני** שהוא כותב.

> **הערה על מסמכים 34 ו-01:** נכתבו סביב Supabase (לפני החלטת ה-MySQL+API).
> ה**סכמה** שבהם עדיין שימושית כ-DDL reference לצוות הפיתוח, אבל חלקי
> RLS/Auth/Storage/Edge-Functions **אינם רלוונטיים לנו** — הם באחריות הצוות.
> מסמך זה (36) + מסמך 37 הם מקור-האמת הארכיטקטוני.
