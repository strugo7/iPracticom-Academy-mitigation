# יישור גרסאות ועקביות — ביקורת כל המסמכים

> **מטרה:** תוצאת ביקורת עקביות על כל 37 המסמכים, ומיפוי מקור-האמת לכל נושא.
> **מעמד:** מסמך-על. קרא אותו כדי לדעת *איזה* מסמך מוסמך *למה*.

---

## 1. תוצאת הביקורת — בקצרה

עברתי על כל 37 המסמכים. **בריפי העיצוב (02-33) תקפים** — העיצוב לא השתנה. מה
שהשתנה הוא **שכבת המימוש** (Supabase → MySQL+API), וזה משפיע רק על סעיפי
ה-handoff/backend, לא על העיצוב. תיקנתי את המסמכים המהותיים והוספתי הערות-גרסה.

**שתי החלטות ששינו הנחות מוקדמות:**
1. **ארכיטקטורה:** MySQL + API פנים-ארגוני (לא Supabase). auth ואחסון = צוות הפיתוח.
2. **שמות שדות:** לפי הדאטה האמיתי (מסמך 35) — `role` לא `system_role`, ועוד.

## 3. תיקוני שמות שדות (מרכזי — מסמך 35 §2.1, §6)

| במסמכים כתוב | הנכון (דאטה אמיתי) |
|---|---|
| `system_role` | **`role`** (ערכים בפועל: user, admin) |
| `created_at` / `updated_at` | `created_date` / `updated_date` |
| `created_by` | `created_by_id` |
| מנהל לפי `role='manager'` | מנהל לפי **`managed_department` מוגדר** |
| `Topic.module_id` | `Topic.shared_module_id` |

> מופיע ב: 01,02,03,09,11,16,26,27,30,34. **אל תתקן ידנית בכל מסמך** — מסמך 35
> הוא מקור-האמת לשמות שדות, וכל שלב בתוכנית 36 מפנה אליו.
