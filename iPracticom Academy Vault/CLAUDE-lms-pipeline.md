מטרת הקובץ: כשהצוות הטכני יתחיל לממש את הפייפליין ואת שכבת הרינדור, יהיה כאן תיעוד מוכן של ההחלטות שכבר התקבלו, כדי לא לאבד אותן בין שיחות.

## חלק א׳ — פייפליין קליטת תוכן (Ingest Pipeline)

**סטטוס: תכנון, טרם מומש.**

### זרימה מתוכננת

```
נציג מעלה קובץ (PDF/מסמך)
  → טופס העלאה במערכת ה-LMS
  → Webhook ל-n8n
  → שמירת המקור הגולמי (immutable) ב-_raw-sources/
  → קריאה ל-Claude API: חילוץ + פירוק אטומי + הצעת domain/vendor/type
  → כתיבה ל-_needs-review/ (כבר קיימת ריקה במבנה הVault הסופי — ראה CLAUDE.md; לא ישירות ל-Vault הפעיל!) עם status: needs-review
  → התראת Telegram/Slack לאישור אנושי (בדומה ל-approval gate שכבר קיים בפייפליין הווידאו)
  → אישור אנושי → מעבר לתיקיית ה-domain/vendor הנכונה, status: reviewed
  → סנכרון ל-Pinecone (support-rag-system) לזמינות מיידית לבוט
  → חזרה למערכת השיעורים עם קישור ל-note החדש
```

### עקרונות מחייבים כשזה ייבנה בפועל

- **שום תוכן לא נכנס ל-Vault הפעיל בלי אישור אנושי.** הסוכן/הפייפליין לא משנה `status` ל-`reviewed` בעצמו — זהה לעיקרון ב-`CLAUDE.md`.
- **המקור הגולמי (`_raw-sources/`) לעולם לא נערך.** אם המקור היה שגוי, מעלים גרסה חדשה, לא עורכים את הישנה.
- **סיווג אוטומטי (`domain`/`vendor`) על תוכן חופשי (PDF/מסמך) פחות מדויק** מסיווג לפי מבנה תיקיות (כמו ב-`convert_vault.py` על JSON מובנה) — לכן שלב האישור האנושי הוא חלק קבוע מהארכיטקטורה, לא שלב זמני שיוסר כשהמערכת "תבשיל".
- אותו webhook שמסנכרן ל-Vault יכול לדחוף גם embedding ל-Pinecone באותה נשימה — אין צורך בשני pipelines נפרדים.

## חלק ב׳ — שכבת רינדור (LMS כ-Headless CMS)

**סטטוס: תכנון, טרם מומש.**

### העיקרון

ה-Vault הוא מקור האמת (Git-based headless CMS). ה-LMS לא "מציג markdown גנרי" — הוא מפרסר את ה-frontmatter, בוחר קומפוננטת רינדור לפי שדה `type`, ומזין את גוף ה-markdown לתוכה.

### מיפוי type → קומפוננטה (הצעה ראשונית, לאשר עם הצוות הטכני)

| `type` | קומפוננטת רינדור |
|---|---|
| `concept` | כרטיס מושג — כותרת + טקסט + "מושגים קשורים" כ-chips לחיצים |
| `interface` | תבנית "סיור במסך" — טקסט לצד תמונה גדולה עם caption מוצג |
| `procedure` | רשימת צעדים ממוספרת, עם progress/checkbox |
| `issue` | תבנית עץ החלטה — שאלה → ענפי תשובה |
| `topology` | דיאגרמת רכיבים (יתכן SVG שנבנה מרשימת ה-vendors שב-note) |
| `moc` | **לא מרונדר למשתמש קצה** — כלי ניווט פנימי לצוות בלבד |

### דרישות טכניות שהצוות יצטרך לפתור

1. **Resolver לתחביר `![[filename]]`** — זה תחביר Obsidian, לא markdown תקני. מנוע רינדור רגיל (react-markdown וכו') לא יפרש אותו כברירת מחדל; צריך plugin/preprocessing שממיר לנתיב/URL אמיתי באחסון של ה-LMS.
2. **ה-caption מתחת לתמונה משמש כפול**: גם ל-RAG (context טקסטואלי) וגם כ-`alt`/הסבר מוצג בפועל למשתמש הקצה. איכות הניסוח בו קריטית משתי סיבות, לא רק אחת.
3. **עיצוב (Brand Kit) חי ברמת ה-theme של ה-LMS, לא בתוך ה-markdown.** צבעים (#0075DB, #2EB4FF, #181D24, #757D86, #FFFFFF, #F4FBFF), פונטים (Ploni/Heebo), RTL — כל אלה קבועים ברמת הקומפוננטה, כדי שאותו Vault יוכל לשרת עיצוב עתידי אחר בלי לגעת בתוכן עצמו.
